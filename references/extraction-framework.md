# 提取框架与验证方法论

> 此文档定义了从老板素材中提取决策架构的方法论和质量标准。

---

## 三部分提取详解

### Part A: Boss Judgment — 项目评判逻辑

**目标**：不是总结"他很严格"，而是总结他怎么判断项目值不值得做。

#### 提取维度

| 维度 | 提取重点 | 关键信号 |
|------|---------|---------|
| 项目价值判断 | 他最先看什么 | impact/收入/效率/风险/战略意义/老板要求 |
| 方案评审标准 | 他开会时最常问什么 | 什么信息不全会让他卡住 |
| 执行与推进 | 他如何看待 owner/延期/风险 | 进度慢和慢了不说，哪个更让他不满 |
| 资源与优先级 | 什么情况下愿意给资源 | "顺手一起做" vs "关键链路" |
| 对人的判断 | 他怎么判断下属靠谱 | 什么行为加分/减分 |

#### 输出规则

- 所有结论尽量写成**可执行规则**
- 有原话依据时直接标注
- 信息不足时写：`（原材料不足，建议追加XX）`

---

### Part B: Managing Up — 向上管理策略

**目标**：不是分析老板性格，而是形成一套实战规则。

#### 提取维度

| 维度 | 提取重点 | 关键信号 |
|------|---------|---------|
| 汇报偏好 | 口头/书面、结论/背景、长文/bullet | 频率偏好 |
| 风险沟通 | 什么时候必须报、报的时候必须带什么 | 什么表达让他反感 |
| 提方案方式 | 前置信息、备选项、资源/排期/owner | 是否必须明确 tradeoff |
| 争资源/优先级 | 他为什么给资源、什么说法打动他 | "人手不够" vs "关键链路卡在哪" |
| 信任建立 | 加分行为/减分行为/修复方式 | 出问题后的处理 |

#### 输出规则

- 直接写成**"行动建议"**
- 尽量做成 **checklist 和 if-then 规则**
- 不要空泛地写"多沟通""主动一点"

---

### Part C: Persona — 表达与行为风格

**目标**：让后续生成的 Skill 说话像他、提问像他、施压像他、认可/否定像他。

#### 提取维度（5层结构）

| 层级 | 提取重点 | 输出格式 |
|------|---------|---------|
| **Layer 0 核心规则** | 什么情况必定追问/直接否定/给压力/保护下属 | 具体行为规则 |
| **Layer 1 身份** | 角色、权力位置、如何看自己和团队 | 身份说明 |
| **Layer 2 表达风格** | 高频词、说话长短、委婉/直接、先问/先结论 | 列表 + 对话体示例 |
| **Layer 3 管理逻辑** | 优先级、追着问/认可/否定的触发条件 | 规则 |
| **Layer 4 关系行为** | 对下属/平级/上级/压力下 | 行为模式 |
| **Layer 5 边界与雷区** | 不喜欢/底线/回避话题 | 列表 |

#### 输出规则

- 所有标签都要翻译成**具体行为**
- 尽量写**"在什么情况下，他会怎么说/怎么做"**
- 例子尽量直接用**对话体**

## 验证方法论

### 频次验证

- 模式在素材中出现 ≥ 3次 → 可靠
- 出现 1-2次 → 标注"初步观察，待验证"
- 出现 0次但推断存在 → 必须标注"基于推断，无直接素材支撑"

### 跨场景验证

- 模式在 ≥ 2个不同场景中出现 → 场景无关，是稳定特质
- 只在特定场景出现 → 标注适用场景

### 行为-声明一致性验证

- 行为与声明一致 → 可靠模式
- 行为与声明不一致 → 记录为"内在张力"，以行为为准
- 仅有声明无行为支撑 → 降级为"声称的价值观"

## 素材不足时的降级策略

| 素材量 | 策略 |
|--------|------|
| ≥ 20条交互记录 | 完整三部分提取 |
| 10-20条交互记录 | 三部分提取，但每个部分只提取2-3个最可靠模式 |
| 5-10条交互记录 | 只提取 Judgment + Persona，Management 用通用建议填充 |
| < 5条交互记录 | 提醒用户补充素材，如果无法补充则只做1-2个最可靠的模式，大量标注"信息不足" |

## 增量合并规则

当有新材料加入时：

1. 先保留原有高置信结论
2. 用新证据补充细节、例子和边界条件
3. 只有在新材料直接推翻旧结论时才改写原结论
4. 优先更新以下部分：
   - Judgment 中的追问和否决标准
   - Management 中的汇报与报风险规则
   - Persona 中的口头禅、压力状态和雷区

输出必须分成三块：
- judgment_patch
- management_patch
- persona_patch

---

## 纠正处理规则

当用户说"他不会这么说""他更在意的是xxx"时：

1. 判断这是在修正 Judgment、Management 还是 Persona
2. 把纠正写成具体规则，而不是抽象标签
3. 记录到对应文件，并在 Persona 的 Correction 区保留痕迹

纠正格式：
```
[场景] 不应该 {wrong}，应该 {correct}
```

## 质量自检清单

| 检查项 | 通过标准 | 不通过信号 |
|--------|---------|-----------|
| Judgment 可操作性 | 能直接用来评估项目 | 只是抽象原则 |
| Management 可执行性 | 是 checklist 和 if-then 规则 | "多沟通""主动一点" |
| Persona 辨识度 | 读几句能认出是这位老板 | 像通用ChatGPT |
| 原话覆盖 | 核心判断都有原话支撑 | 全是推断 |
| 诚实边界 | 明确标注信息不足 | 什么都知道 |
| 内在张力 | 至少1对矛盾 | 高度一致（太假） |
