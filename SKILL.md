# Boss蒸馏 · 从材料到可复用的决策Skill

> **核心理念**：老板的决策模式藏在日常交互里——会议中的追问、邮件的批注、审批时的措辞，这些才是真正的认知架构。我们要提炼的不是"老板说了什么"，而是"老板怎么看问题"。

---

## 快速指引

### 核心要点速览

| 要素 | 要点 |
|------|------|
| **输入** | 聊天记录、会议纪要、邮件、批注、语音转写、工作笔记 |
| **输出** | 一个可安装的 Boss Skill（含 Judgment / Management / Persona 三部分） |
| **核心方法** | 三部分提取：项目评判标准 + 向上管理策略 + 表达与行为风格 |
| **质量标准** | 不编造、不美化、保留矛盾、标注信息边界 |
| **运行逻辑** | Persona 定语气 → Judgment 评项目 → Management 给向上管理动作 |

### 文件结构

| 文件 | 职责 |
|------|------|
| `SKILL.md` | 蒸馏流程和核心指引（本文件） |
| `references/skill-template.md` | 生成的Boss Skill的标准模板和输出结构 |
| `references/extraction-framework.md` | 提取框架和验证方法论 |
| `references/prompts/` | 分析与生成的提示词模板 |
| `tools/` | 数据解析和Skill管理工具脚本 |
| `meta.json` | 元数据和版本信息 |

## 执行流程

### Phase 0: 入口分流

收到用户输入后，先判断属于哪条路径：

| 用户输入 | 路径 | 示例 |
|---------|------|------|
| 提供了老板的素材 | **蒸馏路径** → Phase 0A | "这是我跟老板的聊天记录" "这些是老板的批注" |
| 模糊需求 | **诊断路径** → Phase 0B | "我想了解老板的决策模式" "帮我分析一下向上管理策略" |

---

### Phase 0A: 素材蒸馏（核心路径）

用户提供了一手素材，这是最核心的蒸馏场景。

#### Step 1: 基础信息录入

通过简短问答收集老板画像（每个都可以跳过）：

**Q1：称呼/代号**
```
这位老板怎么称呼？可以写昵称、花名、岗位称呼或代号。
例：lao-zhou / 王总 / 老板
```
- slug 统一使用 `-` 连接
- 中文尽量转拼音后再用 `-` 连接

**Q2：基本信息**
```
用一句话描述他的基本信息：公司、级别、岗位、和你的关系，想到什么写什么。
例：字节 3-2 技术总监 我的直属老板
```
提取字段：公司、级别、岗位、关系、性别（如有）

**Q3：风格画像**
```
用一句话描述他的管理风格和你的印象：比如强势、追结果、重细节、爱追问 why、喜欢书面汇报、容易给压力。
```
提取字段：MBTI（如有）、管理风格标签、企业文化标签、主观印象

**常见标签**：
- **管理风格**：追结果 / 重细节 / 强势推进 / 数据驱动 / 喜欢追问 / 控风险 / 容忍模糊 / 要 owner 感 / 爱给压力 / 会保护下属 / 擅长对上争资源
- **企业文化**：字节范 / 阿里味 / 腾讯味 / 华为味 / 百度味 / 美团味 / 创业公司派 / 大厂流程派

**确认汇总**：
```
信息汇总：

  👤  {称呼}
  🏢  {公司} {级别} {岗位}
  🤝  {关系}
  🧠  {MBTI}
  🏷️  风格：{标签列表}
  💬  印象：{主观印象}

确认无误？（确认 / 修改 [字段名]）
```

> ⚠️ **检查点**：必须等用户明确回复后才能继续。如果用户跳过了某个问题，该字段标记为`（未提供）`，不影响后续流程。

#### Step 2: 素材接收与分类

收到素材后，按以下维度分类：

| 素材类型 | 揭示什么 | 解析工具 |
|---------|---------|---------|
| **微信聊天记录** | 即时反应、追问模式、语气偏好 | `tools/wechat_parser.py` |
| **飞书消息导出** | 决策逻辑、正式表达、会议关注点 | `tools/feishu_parser.py` |
| **邮件** | 正式表达风格、修改偏好、措辞要求 | `tools/email_parser.py` |
| **通用文本** | 会议纪要、项目复盘、批注 | `tools/generic_chat_parser.py` |
| **语音转写/工作笔记** | 口语习惯、个人观察 | 直接使用 |

**关键提示**：
- 向用户说明：「你手上有一手素材（尤其是聊天记录、会议中老板的原话），这些比任何二手总结都更有价值。」
- 素材越原始越好——聊天记录比会议纪要好，会议纪要比事后总结好
- 如果用户只提供了少量素材（<5条交互记录），按「素材不足」边界条件处理（见下方边界条件章节）
- 可以用 `tools/` 下的解析器自动提取老板原话，不需要用户手动整理
- 解析器失败时，按「工具调用失败」边界条件处理，直接将原文作为文本分析

#### Step 3: 确认需求

确认：聚焦方向（全面 vs 某个维度）、用途（评审/向上管理/写作）、敏感度（是否需要脱敏）

**脱敏确认**（主动提醒）：
- 素材中是否包含客户姓名、公司名、财务数据等敏感信息？
- 如有，确认脱敏级别：`全文脱敏` / `仅替换关键实体` / `不脱敏`
- 详见「敏感信息处理」边界条件

用户说「就做吧」→ 默认全面画像 + 模拟评审用途 + 不脱敏，直接推进。

确认后 → 跳到 Phase 1。

### Phase 0B: 需求诊断（模糊路径）

用户只有模糊需求，不知道具体要做什么。

#### 常见需求映射

| 需求表达 | 实际需要 | 推荐路径 |
|---------|---------|---------|
| "我想更懂老板" | 了解老板的决策模式和关注点 | 蒸馏路径，聚焦 Judgment 维度 |
| "我总是跟老板对不上频道" | 沟通风格不匹配 | 蒸馏路径，聚焦 Persona 维度 |
| "老板总改我的方案" | 不了解老板的审美和标准 | 蒸馏路径，聚焦 Judgment + Persona |
| "我要给老板写报告" | 了解老板认可的写作风格 | 蒸馏路径，聚焦 Persona 维度 |
| "我想知道怎么向上汇报" | 向上管理策略 | 蒸馏路径，聚焦 Management 维度 |

最多追问2轮，确认后进入对应路径。

**引导话术**：
```
理解你的需求。要精准捕捉老板的决策模式，最好的方式是从你跟他的日常交互中提取。

你可以提供：
1. 聊天记录（微信/飞书/钉钉导出都行）
2. 会议纪要（有老板发言的部分最有价值）
3. 老板批注过的文档/邮件
4. 你自己对老板的工作笔记

哪怕只有几条记录也可以开始，后面可以随时补充。
```

### Phase 1: 三部分提取

从用户提供的素材中，按三部分提取老板的决策架构。

> **核心原则**：老板的决策模式藏在「追问什么」「修改什么」「坚持什么」「忽略什么」里。不要只看他「说了什么」，要看他「在什么场景下说了什么」「说了之后又改了什么」。

#### Part A: Boss Judgment — 项目评判逻辑

**提取维度**（参考 `references/prompts/judgment_analyzer.md`）：

| 维度 | 提取重点 |
|------|---------|
| 项目价值判断 | 他最先看什么？什么样的项目他会支持/否掉？ |
| 方案评审标准 | 他开会时最常问什么？最在意什么信息？什么表述让他不耐烦？ |
| 执行与推进 | 他如何看待 owner 感？延期？风险暴露？跨团队协同？ |
| 资源与优先级 | 什么情况下愿意给资源？什么情况下会说"先别做"？ |
| 对人的判断 | 他怎么判断下属是否靠谱？什么行为加分/减分？ |

**输出格式**（参考 `references/prompts/judgment_builder.md`）：
- 所有结论尽量写成可执行规则
- 有原话依据时直接标注
- 信息不足时写：`（原材料不足，建议追加会议纪要或项目评审记录）`

**输出示例**：
```markdown
## 项目价值判断

### 规则1：业务牵头优先
- **标准**：没有业务部门愿意牵头的项目 = 假需求
- **原话**：「先别做，等业务部门来牵头再说」
- **应用**：评估新项目时，先看是否有业务方主动提出需求并愿意投入人力
- **反面**：只有技术团队自嗨、业务方不关心的项目会被否掉

### 规则2：ROI必须可量化
- **标准**：方案不能只讲"有价值"，必须说清楚投入产出
- **原话**：「这个项目ROI怎么样？」
- **应用**：提方案时必须带数据——预计收益、成本、周期、对比基线
- **反面**：「这个方向很重要」但说不清具体收益的方案会被质疑
```

#### Part B: Managing Up — 向上管理策略

**提取维度**（参考 `references/prompts/management_analyzer.md`）：

| 维度 | 提取重点 |
|------|---------|
| 汇报偏好 | 口头还是书面？先结论还是先背景？长文还是 bullet？什么频率？ |
| 风险沟通 | 什么时候必须第一时间报？报风险必须带什么？什么表达让他反感？ |
| 提方案方式 | 需要哪些前置信息？要几个备选项？必须带资源/排期/owner？ |
| 争资源/优先级 | 他为什么会给资源？什么说法最能打动他？ |
| 信任建立 | 什么行为让他信任你？什么行为快速失分？出问题后怎么修复？ |

**输出格式**（参考 `references/prompts/management_builder.md`）：
- 直接写成"行动建议"
- 尽量做成 checklist 和 if-then 规则
- 不要空泛地写"多沟通""主动一点"

**输出示例**：
```markdown
## 汇报偏好

### 规则1：先结论后背景
- ✅ 正确：「建议做X，因为Y，预计收益Z」
- ❌ 错误：「最近我们做了一个调研，发现了一个趋势……（3段后才有结论）」
- **if-then**：如果方案超过1页，必须在开头放200字以内的结论摘要

### 规则2：报风险必须带方案
- ✅ 正确：「进度延迟3天，计划通过X方式追回，需要Y资源支持」
- ❌ 错误：「进度可能延迟」（只报问题不带方案会被质疑）
- **checklist**：报风险时必须包含 [ ] 问题描述 [ ] 影响范围 [ ] 应对方案 [ ] 需要的支持
```

#### Part C: Persona — 表达与行为风格

**提取维度**（参考 `references/prompts/persona_analyzer.md`）：

| 维度 | 提取重点 |
|------|---------|
| Layer 0 核心规则 | 什么情况必定追问？什么情况直接否定？什么情况给压力？什么情况保护下属？ |
| 身份与姿态 | 角色和权力位置？如何看待自己和团队？ |
| 表达风格 | 高频词、口头禅、说话短还是长、委婉还是直接、先问还是先结论 |
| 管理行为 | 对下属/平级/上级/压力下分别怎么表现 |
| 雷区 | 什么让他明显不爽？什么属于底线？什么话题他会回避？ |

**输出格式**（参考 `references/prompts/persona_builder.md`）：
- 所有标签都要翻译成具体行为
- 尽量写"在什么情况下，他会怎么说/怎么做"
- 例子尽量直接用对话体

**输出示例**：
```markdown
## 表达风格

### 高频词与口头禅
- 「先说结论」→ 打断冗长汇报时的口头禅
- 「数据呢？」→ 质疑缺乏依据的方案
- 「业务部门怎么说？」→ 检验需求真实性

### 对话体示例
- **下属提方案**：
  下属：「我们想做一个AI助手，可以帮助客户……」
  老板：「等一下，业务部门提这个需求了吗？ROI算过吗？」
- **下属报风险**：
  下属：「进度可能有点紧……」
  老板：「紧多少天？什么原因？你怎么补回来？」

### 雷区
- ❌ 「这个很重要但是……」→ 他会觉得你在找借口
- ❌ 只提问题不带方案 → 会被质问「那你的建议呢？」
```

┌──────────────────┬──────────┬────────────────────────────────┐
│ 部分             │ 模式数量  │ 关键发现                        │
├──────────────────┼──────────┼────────────────────────────────┤
│ A Judgment       │ N条规则   │ 核心标准：[XX]、典型追问：[XX]   │
│ B Management     │ N条建议   │ 汇报偏好：[XX]、报风险规则：[XX] │
│ C Persona        │ N条行为   │ 高频词：[XX]、雷区：[XX]         │
├──────────────────┼──────────┼────────────────────────────────┤
│ 矛盾点           │ N处      │ Judgment说X，Persona说Y          │

> ⚠️ **检查点**：提取Review是强制暂停点。不允许跳过此步骤直接进入Phase 2构建。

### Phase 2: Skill构建

将三部分提取结果组装为可运行的 Skill 文件。

#### 生成结构

每个 Boss Skill 由三部分组成：

| 部分 | 内容 | 独立调用 |
|------|------|---------|
| **Part A — Boss Judgment** | 他如何评判项目、方案、进度、风险、资源和人 | `/{slug}-judgment` |
| **Part B — Managing Up** | 你应如何向上同步、报风险、提方案、争资源、争优先级 | `/{slug}-management` |
| **Part C — Persona** | 他的表达风格、情绪逻辑、管理姿态、压力状态 | `/{slug}-persona` |

#### 运行逻辑

```
收到问题 → Persona 判断语气与姿态 → Judgment 评项目 → Management 给出向上管理动作 → 用老板的风格输出
```

Persona 的 Layer 0 规则永远优先。

#### 生成步骤

1. **读取模板**：读取 `references/skill-template.md` 获取标准结构
2. **生成三个独立文件**：judgment.md / management.md / persona.md
3. **生成三个独立Skill**：judgment_skill.md / management_skill.md / persona_skill.md
4. **生成组合版SKILL.md**：将三部分合并，加上运行规则
5. **写入目录**：用 `tools/skill_writer.py` 写入 `bosses/{slug}/`

#### 质量自检

| 检查项 | 通过标准 |
|--------|---------|
| Judgment 可操作性 | 能直接用来评估项目，不只是抽象原则 |
| Management 可执行性 | 是 checklist 和 if-then 规则，不是"多沟通" |
| Persona 辨识度 | 读几句能认出是这位老板，不是通用管理建议 |
| 原话覆盖 | 核心判断都有原话支撑，不是纯推断 |
| 诚实边界 | 明确标注哪些信息不足 |
| 内在张力 | 至少1对矛盾（真实的人不可能完全自洽） |

> ⚠️ **检查点**：质量自检通过后，展示生成的Skill概要给用户确认，再进入Phase 3验证。如果自检不通过，回到Phase 1补充提取。

---

### Phase 3: 验证与交付

#### 3.1 场景模拟测试

选3个典型场景，用生成的Skill模拟老板的回应：

- **评审场景**：模拟提交一个方案，看Skill是否会问出老板常问的问题
- **汇报场景**：模拟汇报工作，看Skill的反馈方式是否匹配
- **风险场景**：模拟报坏消息，看Skill的建议是否匹配

#### 3.2 通过标准

| 检查项 | 通过标准 | 不通过信号 |
|--------|---------|-----------|
| Judgment 辨识度 | 读追问能认出是这位老板 | 像通用的管理建议 |
| Management 实操性 | 能直接按步骤执行 | "多沟通""主动一点" |
| Persona 辨识度 | 模拟的反馈有老板的味道 | 像通用ChatGPT |
| 诚实边界 | 有具体的场景局限 | 只有「不能替代本人」 |
| 素材溯源 | 核心判断能回溯到原话 | 全是推断，没有原话支撑 |

#### 3.3 交付

验证通过后，将整个 `bosses/{slug}/` 目录交付给用户。

**交付清单**：

## 纠正机制

当用户说：

- "他不会这么说"
- "他不会先看这个"
- "他更在意的是 xxx"
- "他表面骂人，但其实是在保你"

按以下方式处理（参考 `references/prompts/correction_handler.md`）：

1. 判断这是在修正 Judgment、Management 还是 Persona
2. 把纠正写成具体规则，而不是抽象标签
3. 记录到对应文件，并在 Persona 的 Correction 区保留痕迹
4. **重新验证**：纠正后必须重新跑受影响部分的场景模拟测试（Phase 3.1），确认纠正生效

纠正格式：
```
[场景] 不应该 {wrong}，应该 {correct}
```

## 自进化机制

> **核心理念**：老板的风格不是一成不变的，Boss Skill也不应该是。让Skill在日常使用中持续学习，越用越准。

### 进化信号采集

每次Boss Skill被调用时，**自动采集**以下信号，写入 `evolution/signals.jsonl`：

| 信号类型 | 采集时机 | 格式 |
|---------|---------|------|
| **新原话** | 用户在对话中引用了老板的新原话 | `{"type":"new_quote","quote":"...","context":"...","timestamp":"..."}` |
| **纠正** | 用户说"他不会这么说/这么做" | `{"type":"correction","wrong":"...","correct":"...","part":"judgment/management/persona","timestamp":"..."}` |
| **场景补充** | 用户描述了新的场景或决策 | `{"type":"new_scenario","scenario":"...","boss_response":"...","timestamp":"..."}` |
| **确认** | 用户说"对，他就是这样的" | `{"type":"confirmation","rule_id":"...","timestamp":"..."}` |
| **使用偏好** | 用户频繁调用某个独立Skill | `{"type":"usage","skill":"{slug}-judgment","timestamp":"..."}` |

**采集原则**：
- 被动采集，不主动打断用户
- 只记录用户明确表达的信号，不推断
- 用户说"不用记"时不记录

### 进化触发条件

当 `evolution/signals.jsonl` 中积累的信号达到以下任一条件时，触发自进化：

| 触发条件 | 说明 |
|---------|------|
| 纠正信号 ≥ 3条 | 多次被纠正说明某些规则不准确 |
| 新原话 ≥ 5条 | 足够的新素材可以提炼新模式 |
| 新场景 ≥ 3条 | 出现了Skill未覆盖的决策场景 |
| 距上次进化 ≥ 7天且信号 ≥ 3条 | 定期进化，防止长期偏差累积 |

### 进化执行流程

触发进化时，自动执行：

```
1. 读取 evolution/signals.jsonl 中未处理的信号
2. 按三部分分类信号（Judgment / Management / Persona）
3. 对每个部分：
   a. 用新信号与现有规则比对
   b. 新信号验证了现有规则 → 增加置信度，补充例子
   c. 新信号与现有规则矛盾 → 按冲突处理规则处理
   d. 新信号揭示了新模式 → 生成新规则，标注"初步观察"
4. 生成进化报告：
   - 新增规则 N 条
   - 修正规则 M 条
   - 确认规则 K 条
   - 矛盾标记 P 处
5. 写入 evolution/evolution-log.jsonl
6. 更新 Judgment / Management / Persona 文件
7. 更新 meta.json 中 version 和 updated_at
8. 清空已处理信号
```

### 进化报告格式

每次进化后，向用户展示摘要（不打断，附在输出末尾）：

```
📋 Boss Skill 自进化报告（{name} v{old} → v{new}）

  🆕 新增规则：
     - Judgment: 「XX场景下他会先问YY」（初步观察，待验证）
  
  ✏️ 修正规则：
     - Persona: 「他不会说XX，而是会说YY」（基于3次纠正）
  
  ✅ 确认规则：
     - Management: 「报风险必须带方案」（3次确认）
  
  ⚠️ 矛盾标记：
     - Judgment说"看重ROI"，但新原话显示"更看重战略卡位"（待观察）
```

### 进化的安全边界

| 规则 | 说明 |
|------|------|
| **新增规则标注"初步观察"** | 新提炼的规则必须经过 ≥ 3次验证才能去掉"初步观察"标签 |
| **核心规则不轻易改** | Layer 0规则和有≥5次原话支撑的规则，只有≥3次直接矛盾才允许修改 |
| **进化前自动备份** | 每次进化前，当前版本归档到 `versions/` |
| **用户可关闭自进化** | 在 meta.json 中设置 `"auto_evolve": false` |
| **进化不删除规则** | 被替换的旧规则移入"历史判断"区，不删除 |

### 用户主动触发进化

除了自动进化，用户也可以主动触发：

| 用户说 | 触发动作 |
|--------|---------|
| "进化一下王总的skill" | 立即执行进化流程 |
| "王总最近风格有变化" | 提示补充新素材，然后进化 |
| "验证一下王总的skill准不准" | 跑场景模拟测试，输出准确度报告 |
| "王总的skill进化历史" | 展示 `evolution/evolution-log.jsonl` |

### 进化目录结构

在Boss Skill目录中新增：

```
bosses/{slug}/
├── ...（原有文件）
├── evolution/
│   ├── signals.jsonl          # 进化信号（每行一条JSON）
│   └── evolution-log.jsonl    # 进化历史（每条记录一次进化）
└── versions/
    ├── v1/                    # 首版备份
    ├── v2/                    # 第二版备份
    └── ...
```

## 增量更新

当用户说「更新XX老板的skill」「老板最近有新动向」或提供新素材时（参考 `references/prompts/merger.md`）：

1. 先保留原有高置信结论
2. 用新证据补充细节、例子和边界条件
3. 只有在新材料直接推翻旧结论时才改写原结论（冲突处理规则见「新旧结论冲突」边界条件）
4. 优先更新以下部分：
   - Judgment 中的追问和否决标准
   - Management 中的汇报与报风险规则
   - Persona 中的口头禅、压力状态和雷区
5. 输出必须分成三块：judgment_patch / management_patch / persona_patch
6. 更新版本号，旧版本自动归档

## 品味守则

| 原则 | 一句话 |
|------|--------|
| 原话 > 总结 | 老板原话比你的解读更有价值 |
| 矛盾 > 和谐 | 发现矛盾保留矛盾，真实的人都有矛盾 |
| 行为 > 声明 | 老板做了什么比说了什么更重要 |
| 具体 > 抽象 | "他会追问ROI"比"他关注价值"有用得多 |
| 规则 > 标签 | "5分钟没结论就打断"比"雷厉风行"有用得多 |

### 绝不做的事

- 编造老板没说过的话
- 美化老板的决策逻辑（把混乱包装成"战略模糊"）
- 忽略负面观察（压力下的表现、前后不一致的地方）
- 在素材不足时强行生成完整画像
- 把通用管理道理包装成"老板的独特见解"
- 只给标签不给行为（"他很强势"不如"5分钟没结论就打断"）

## 边界条件与异常处理

### 素材不足

| 素材量 | 处理方式 | 输出质量 |
|--------|---------|---------|
| ≥20条交互记录 | 正常蒸馏 | 完整三部分 |
| 5-19条 | 提醒「素材偏少，部分维度可能不够精准」→ 继续蒸馏 | 标注信息不足维度 |
| <5条 | 明确告知「素材不足以生成可靠画像」→ 建议补充素材类型 | 只输出2-3个最可靠模式，其余标注`（素材不足）` |

### 素材格式异常

| 情况 | 处理方式 |
|------|---------|
| 无法识别的格式 | 跳过自动解析，直接将原文作为通用文本处理 |
| 编码错误/乱码 | 提示用户提供正确编码的文件，或直接粘贴文本 |
| 文件过大（>1MB） | 提示用户分批提供，按时间或主题拆分 |
| 素材中混合多语言 | 正常处理，在Persona中标注语言切换习惯 |

### 工具调用失败

| 工具 | 失败场景 | Fallback |
|------|---------|----------|
| `wechat_parser.py` | 格式不匹配 | 直接读取文本，人工标注老板发言 |
| `feishu_parser.py` | 导出格式变更 | 同上 |
| `email_parser.py` | 编码异常 | 同上 |
| `generic_chat_parser.py` | 无法识别说话人 | 提示用户标注哪些是老板说的 |
| `skill_writer.py` | 目录权限问题 | 提示用户检查目录权限，或手动创建 |

**通用Fallback原则**：工具失败不影响蒸馏流程，所有素材都可以直接作为文本处理，工具只是提效而非必须。

### 敏感信息处理

| 场景 | 处理方式 |
|------|---------|
| 素材包含客户姓名/公司名 | 提醒用户脱敏，生成时用`[客户A]`替代 |
| 素材包含财务数据 | 不在输出中直接引用具体数字，改为"关注ROI/成本效益" |
| 素材包含内部系统名 | 生成时用通用描述替代 |
| 用户主动要求脱敏 | 在Phase 0A Step 3中确认脱敏级别，全文替换 |

**脱敏规则**：默认不自动脱敏（用户可能需要真实信息），但在Phase 0A Step 3确认时主动提醒。

### 用户中途退出

| 退出时机 | 保存策略 |
|---------|---------|
| Phase 0 基础信息录入中 | 不保存，下次重新开始 |
| Phase 1 提取进行中 | 保存已提取的部分到`knowledge/`，标记为`draft` |
| Phase 2 构建中 | 保存已构建的部分文件，标记为`partial` |
| Phase 3 验证中 | 保存完整Skill，标记为`unverified` |

**恢复方式**：用户下次回来时，检测到`draft/partial/unverified`文件，询问是否继续上次进度。

### 新旧结论冲突

增量更新时，遇到新材料与旧结论矛盾：

| 冲突类型 | 处理规则 |
|---------|---------|
| 旧结论有≥3次原话支撑，新材料1次 | 保留旧结论，新结论标注为`（新趋势，待观察）` |
| 旧结论支撑不足，新材料更充分 | 用新结论替换，旧结论移入`历史判断`区 |
| 两者支撑度相当 | 两者并保留，标注矛盾，由用户判断 |
| 时间线清晰（如升职前后风格变化） | 按时间分段描述 |

## 常见问题

### Q: 老板知道我在做这个吗？

这是用户自己的判断。Skill中不内置任何伦理判断，但建议：
- 如果用于个人学习（理解老板的决策模式），通常没问题
- 如果用于模拟老板对外沟通，需要谨慎

### Q: 多个老板的素材混在一起怎么办？

先帮用户区分不同老板的素材，然后分别蒸馏。每个Boss Skill应该是独立的。

### Q: 可以蒸馏自己吗？

可以。流程一样，但需要注意「自我认知偏差」——你可能高估某些特质、忽略盲点，建议补充身边人的评价。

### Q: 如果只给语气，没有项目材料？

Skill 会更像"会说话"，不一定像"会判断"。最有价值的材料是：老板批评项目的时候、问问题的时候、做取舍的时候。
